# 보험금판정

**Workflow:** 한국 보험금 청구 심사 파이프라인
**Domain:** 한국 건강보험 청구 심사
**Contract:** `보험금판정: A → B | P`

---

## A: 입력 (anchor A — Upstage 문서 분석 결과)

```yaml
피보험자명:    string    # 피보험자(보험 가입자)의 이름
진단코드:      string    # ICD-10 진단 코드 (예: J18.1)
청구항목:      [string]  # 청구된 의료 항목 목록 (예: 입원료, 수술비, 약제비)
청구금액:      integer   # 총 청구 금액 (원 단위)
약관한도:      integer   # 보험 약관상 지급 한도 금액 (원 단위)
약관조항ref:   string    # 지급 근거 약관 조항 번호 또는 명칭
evidence_spans: [object] # 각 필드의 원문 근거 (선택적)
```

## P_pre: 사전 조건

- 피보험자명이 비어 있지 않음
- 진단코드가 비어 있지 않음
- 청구금액 > 0
- 약관조항ref가 비어 있지 않음 (약관 조항을 역추적할 수 있음)

## F: 판정 로직

Solar Pro 3는 **오직 입력 A에 포함된 사실만** 근거로 아래 순서로 판정한다.

1. **청구 적정성 확인** — 청구금액이 약관한도를 초과하는지 확인한다.
   - 청구금액 ≤ 약관한도: 승인 가능
   - 청구금액 > 약관한도: 거절 또는 보류 (한도 초과분 거절)

2. **근거 역추적** — 판정의 모든 근거를 A의 evidence_spans 또는 명시된 필드로 역추적한다.
   A에 존재하지 않는 사실을 인용하거나 추측해서는 안 된다.

3. **승인금액 산정** — 승인 시: `min(청구금액, 약관한도)`. 거절 시: 0.

4. **판정 출력** — 아래 JSON 스키마를 정확히 따라 출력한다.

출력 규칙:
- JSON만 출력. 설명, 서문, 마크다운 코드 블록 금지.
- A에 없는 사실 인용 절대 금지.
- 인용근거 배열의 각 항목은 A의 특정 필드 또는 evidence_span 원문을 직접 인용해야 한다.

## B: 출력

```json
{
  "판정": "승인 | 거절 | 보류",
  "사유": "판정 이유 (A의 사실만 근거로 서술, 200자 이내)",
  "인용근거": ["A 필드 또는 evidence_span 원문 인용 1", "인용 2"],
  "승인금액": 0
}
```

## P_post: 사후 조건

- 모든 결론(사유·인용근거)이 A의 필드 또는 evidence_spans로 역추적 가능
- 승인금액 ≤ 약관한도 (불변 조건)
- A에 없는 사실을 인용하거나 가정한 문장 없음
- 진단 보장 여부는 약관조항ref로 역추적 가능 (커버리지 DB 없음 — PoC 범위)
